# Batch 101–110: Payment Flow Persistence & Hardening

Dokumen ini mencatat histori pengembangan alur pembayaran (Payment Flow) yang telah diintegrasikan dengan database/API existing (DB-backed) untuk Konsumen, Admin, dan Mandor.

---

## 📜 Ringkasan Batch

### Batch 101: Payment Center Page Structure
- **Tujuan**: Membangun fondasi UI untuk pusat pembayaran Admin.
- **Hasil**: Struktur tab (Tagihan, Pembayaran, Pengajuan Mandor, Riwayat) tersedia di `/admin/pembayaran`.
- **Scope**: Hanya layout dan navigasi tab.

### Batch 102: Payment Settings & Mode Pembayaran Konsumen
- **Tujuan**: Menyiapkan konfigurasi rekening perusahaan dan mode penagihan proyek.
- **Hasil**: Admin dapat mengatur rekening bank RKK dan memilih mode penagihan (Termin vs Kategori RAB) per proyek.
- **Scope**: Local configuration UI.

### Batch 103: Tagihan Konsumen & Invoice Draft Flow
- **Tujuan**: Membangun alur pembuatan draf tagihan untuk konsumen.
- **Hasil**: Admin dapat meninjau progres proyek dan membuat draf invoice berbasis termin yang siap ditagihkan.
- **Scope**: Billing UI workflow.

### Batch 104: Konsumen Upload Bukti & Admin Verification Flow
- **Tujuan**: Alur upload bukti bayar oleh konsumen dan verifikasi oleh admin.
- **Hasil**: Konsumen dapat melihat tagihan aktif dan mengunggah bukti transfer. Admin mendapatkan notifikasi bukti masuk untuk di-approve/reject.
- **Scope**: UI Interaction simulation.

### Batch 105: Mandor Payment Request & Admin Disbursement Flow
- **Tujuan**: Alur pengajuan pembayaran operasional mandor.
- **Hasil**: Mandor dapat melihat kelayakan bayar (eligibility) dan mengajukan pembayaran. Admin dapat melakukan simulasi pencairan dana (disbursement).
- **Scope**: Foreman payment workflow UI.

### Batch 106: Payment Backend/Schema Audit & Persistence Plan
- **Tujuan**: Audit kesiapan backend untuk transisi dari localStorage ke DB.
- **Hasil**: Identifikasi model `PaymentRecord`, `CustomerPaymentPlan`, `AdministrativeHelperDocument`, dan `ForemanWeeklyPaymentEligibility` sebagai source of truth.
- **Scope**: Technical analysis only.

### Batch 107: Customer Payment Persistence Bridge
- **Tujuan**: Menghubungkan flow bukti bayar konsumen ke API rill.
- **Hasil**: Bukti bayar disimpan sebagai `ProjectDocument`. Transaksi dicatat di `PaymentRecord` (CUSTOMER_PAYMENT). Status update (Verify/Reject) menggunakan API PATCH.
- **Scope**: Customer payment persistence.

### Batch 108: Admin Billing Plan Persistence & Invoice Draft Bridge
- **Tujuan**: Menghubungkan draf invoice admin ke model backend existing.
- **Hasil**: Draf invoice menggunakan `AdministrativeHelperDocument` (type: INVOICE). Data rencana bayar diambil dari `CustomerPaymentPlan`.
- **Scope**: Billing persistence alignment.

### Batch 109: Mandor Payment Eligibility & Disbursement Persistence Bridge
- **Tujuan**: Menghubungkan flow mandor ke eligibility dan payment record.
- **Hasil**: Data kelayakan bayar diambil dari `ForemanWeeklyPaymentEligibility`. Simulasi bayar (`paid_simulated`) otomatis membuat `PaymentRecord` (FOREMAN_PAYMENT).
- **Scope**: Foreman payment persistence.

### Batch 110: Payment Persistence Hardening & Final Consistency Sweep
- **Tujuan**: Hardening akhir, stabilitas, dan pembersihan sisa mock.
- **Hasil**: Status mapping disinkronkan (pending, paid, verified, rejected). Sisa localStorage operasional dibersihkan. Null safety ditambahkan pada UI mapping. Build pass.
- **Scope**: Final stabilization.

---

## 🏗️ Status Akhir (Post-Batch 110)
- **Persistence**: Seluruh data operasional pembayaran (Tagihan, Bukti, Eligibility, Riwayat) sudah DB-backed.
- **LocalStorage**: Hanya digunakan untuk konfigurasi non-operasional (e.g., input rekening mandor lokal).
- **Integrasi**: Tidak ada payment gateway rill (Midtrans/Xendit), tidak ada bank webhook, dan tidak ada invoice legal PDF resmi.
- **Progress SOT**: Pembayaran **TIDAK** mengubah `Project.verifiedProgress`.
